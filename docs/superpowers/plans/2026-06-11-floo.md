# floo Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `agents-support` as a clean-slate, self-hostable remote-console tool — `floo` (client) + `floo-powder` (operator) — in a new repo `interkelstar/floo`, generalized off agents-deployed, with a real CLI, cross-distro relay, and DoS hardening, while keeping the proven trust model intact.

**Architecture:** Copy the working, tested `agents-support` tree into a new repo, then transform it: rename the binaries/refs, replace the agents-deployed-specific identity with a generic resolver + optional hook, add a client config store + proper CLI, add `floo-powder init`, generalize the relay installer across distros, and add relay DoS hardening. agents-deployed becomes the first *consumer* (its botname logic ships as an identity hook, zero agents-deployed code in core).

**Tech Stack:** bash, OpenSSH (CA certs, reverse stream-local forward), python3 (port pick, PTY recorder, recording renderer), nc, systemd, distro package managers, firewalld/ufw/nftables, fail2ban, bats-less custom test harness (carried over).

**Source of truth:** `~/projects/agents-support` (current, tested: 44-assertion suite green). Spec: `docs/superpowers/specs/2026-06-11-floo-generalization-design.md`.
